## 1. Domaines & pondérations de l'examen

|#|Domaine|Pondération|
|---|---|---|
|1|Sécuriser les identités et l'accès|15–20 %|
|2|Sécuriser le réseau|20–25 %|
|3|Sécuriser le calcul, le stockage et les bases de données|20–25 %|
|4|Sécuriser Azure avec **Defender for Cloud** et **Microsoft Sentinel**|**30–35 %**|

➡️ Le domaine 4 pèse le plus. Donnez-lui un poids équivalent dans votre planning.

---

## 2. Pré-requis recommandés

Avant d'attaquer AZ-500, assurez-vous d'avoir :

- Une connaissance pratique d'Azure (équivalent **AZ-104** ou expérience opérationnelle équivalente).
- Les bases de la sécurité informatique (chiffrement, IAM, réseau, journalisation).
- Un compte Azure actif :
    - **Compte gratuit Azure** (200 USD de crédit, 30 jours) — suffisant pour la plupart des labs.
    - Ou un **abonnement Pay-As-You-Go** avec budget mensuel < 30 USD si vous étalez sur 12 semaines.
- Outils installés : **Azure CLI**, **Azure PowerShell (Az module)**, **VS Code** + extensions Azure, **Cloud Shell**.

## 3. Plan en 12 semaines (≈ 8–10 h/semaine)

### Phase 1 — Préparation & fondations 

**Objectif :** poser le cadre et combler les lacunes Azure.

- [ ] Lire le **Study Guide officiel** AZ-500 sur Microsoft Learn (en entier).
- [ ] Créer / vérifier le compte Azure et un **resource group dédié au lab** (`rg-az500-lab`).
- [ ] Installer Azure CLI, Az PowerShell, VS Code, Bicep CLI.
- [ ] Réviser les bases AZ-104 si besoin : ARM/Bicep, IAM Azure, ressources de base.
- [ ] Suivre le parcours d'apprentissage **« Microsoft Azure Security Engineer (AZ-500) »** sur Microsoft Learn (modules d'introduction).
- [ ] Constituer votre **fiche de glossaire** (KQL, NSG, PIM, MDC, Sentinel, etc.).

**Livrables :** environnement de lab fonctionnel + glossaire personnel.

---

### Phase 2 — Domaine 1 : Identité & accès 

**Objectif :** maîtriser Microsoft Entra ID et la gouvernance des accès.

**Théorie :**

- Microsoft Entra ID (anciennement Azure AD) : tenants, utilisateurs, groupes, rôles.
- Authentification : MFA, méthodes passwordless, FIDO2, Windows Hello.
- **Conditional Access** : signaux, conditions, contrôles, sessions.
- **RBAC** : rôles built-in vs custom, attribution, héritage.
- **PIM (Privileged Identity Management)** : rôles éligibles vs assignés, activation, reviews.
- **Identity Protection** : risk users, risk sign-ins, policies de remédiation.
- **Identités managées** (system-assigned, user-assigned), service principals, app registrations.
- **Entra ID B2B / B2C**, Cross-tenant access settings.

**Labs (à faire les mains dans le clavier) :**

- [ ] Configurer une **Conditional Access policy** qui exige MFA pour les admins hors réseau d'entreprise.
- [ ] Créer un **rôle RBAC custom** limité à la lecture de Storage et l'attribuer.
- [ ] Activer **PIM** sur un rôle, faire une activation just-in-time, créer une access review.
- [ ] Déployer une fonction Azure utilisant une **identité managée** pour lire un secret depuis Key Vault.

**Ressources clés :** parcours Microsoft Learn « Manage identity and access for Microsoft Azure ».

---

### Phase 3 — Domaine 2 : Réseau sécurisé (Semaines 5-6)

**Objectif :** sécuriser le réseau Azure de bout en bout.

**Théorie :**

- **VNet, subnet, peering, hub-spoke**.
- **NSG / ASG** : règles, priorités, application.
- **Azure Firewall** (Standard/Premium) : DNAT, threat intel, IDPS.
- **Web Application Firewall** sur Application Gateway / Front Door.
- **Private Link / Private Endpoints** vs Service Endpoints (différences clés !).
- **DDoS Protection** : Standard vs Network plans.
- **Azure Bastion** (vs JIT VM access).
- **VPN Gateway, ExpressRoute** : sécurité et chiffrement.
- DNS privé Azure, **Azure DNS** sécurisé.

**Labs :**

- [ ] Déployer une topologie hub-spoke avec un Azure Firewall en hub.
- [ ] Configurer une NSG avec **journalisation des flux (NSG Flow Logs)** vers Log Analytics.
- [ ] Mettre une App Service derrière un **Private Endpoint** + résolution DNS privée.
- [ ] Déployer un WAF sur Application Gateway et tester les règles OWASP.
- [ ] Activer Bastion et désactiver les IP publiques sur les VM.

**Pièges fréquents :** différencier Service Endpoint (côté service, par sous-réseau) et Private Endpoint (NIC dans votre VNet, IP privée). Comprendre l'ordre de priorité des règles NSG.

---

### Phase 4 — Domaine 3 : Compute, Storage, Databases (Semaines 7-8)

**Objectif :** durcir les charges de travail.

**Théorie — Compute :**

- Hardening VM : Azure Disk Encryption (ADE), Trusted Launch, Confidential VMs.
- **AKS** : Microsoft Defender for Containers, RBAC AKS, Pod Identity, network policies, image scanning.
- **ACR** : Content Trust, Defender for Containers, scan Trivy/Microsoft.
- **App Service** : managed identity, deployment slots, restrictions IP, SCM access.
- **Update Management**, **Endpoint Protection**, **JIT VM access**.

**Théorie — Stockage :**

- **Azure Storage** : SSE, **CMK** avec Key Vault, infrastructure encryption.
- **SAS tokens** (account vs service vs user delegation), stored access policies.
- Firewall et VNet rules sur Storage, immutability, soft delete, versioning.

**Théorie — Bases de données :**

- **Azure SQL** : TDE, Always Encrypted, Dynamic Data Masking, Auditing, Defender for SQL, Vulnerability Assessment.
- **Cosmos DB** : RBAC data plane, IP firewall, encryption, customer-managed keys.

**Théorie — Secrets :**

- **Key Vault** : RBAC vs access policies (différence cruciale !), soft delete, purge protection, HSM, rotation, references depuis App Service / VM.

**Labs :**

- [ ] Créer un Key Vault avec **purge protection** et y référencer un secret depuis une App Service via identité managée.
- [ ] Activer Defender for SQL et générer un Vulnerability Assessment.
- [ ] Déployer un cluster AKS avec **Microsoft Entra integration** + Azure Policy add-on.
- [ ] Configurer SSE avec CMK sur un Storage Account.
- [ ] Générer un **User Delegation SAS** depuis une identité Entra ID.

---

### Phase 5 — Domaine 4 : Defender for Cloud & Sentinel (Semaines 9-10)

**⚠️ Domaine le plus lourd (30-35 %). Y consacrer du temps de qualité.**

**Théorie — Microsoft Defender for Cloud (MDC) :**

- **CSPM** (Cloud Security Posture Management) : Foundational vs Defender CSPM.
- **CWPP** (Cloud Workload Protection) : plans Defender (Servers, Storage, SQL, Containers, Key Vault, App Service, DNS, Resource Manager).
- **Secure Score** : recommandations, contrôles, remédiation manuelle vs auto.
- **Regulatory compliance** : ISO 27001, PCI-DSS, NIST, custom standards.
- **Workflow automation**, governance rules, **Just-In-Time VM Access**, **adaptive application controls**.

**Théorie — Microsoft Sentinel :**

- Workspace Log Analytics + Sentinel (différence avec MDC).
- **Data connectors** (Azure Activity, AAD, Office 365, AWS, syslog, CEF, custom).
- **KQL** : à pratiquer sérieusement (`where`, `summarize`, `join`, `parse`, `extend`, `make-series`).
- **Analytics rules** : scheduled, NRT, Microsoft, Fusion, anomaly.
- **Incidents**, **investigation**, **entities**.
- **Playbooks** (Logic Apps), automation rules.
- **Workbooks**, hunting queries, threat intelligence, watchlists, UEBA.

**Labs :**

- [ ] Activer **MDC** + 2-3 plans Defender et observer les recommandations.
- [ ] Déployer une **logique de remédiation automatique** (Workflow Automation).
- [ ] Onboarder Sentinel, connecter au moins 3 data connectors (AAD sign-in, Activity, Defender for Cloud).
- [ ] Écrire 3 requêtes **KQL** d'investigation et les sauvegarder en hunting queries.
- [ ] Créer une **analytics rule** scheduled qui détecte un brute-force RDP, et un playbook qui désactive l'utilisateur.

**KQL — exemples à savoir reconnaître/écrire :**

```kql
SigninLogs
| where ResultType != 0
| summarize FailedAttempts = count() by UserPrincipalName, IPAddress
| where FailedAttempts > 10
```

## 5. Conseils stratégiques

1. **Faites les labs.** L'examen comporte des questions très pratiques (ex : ordre exact de création d'un Private Endpoint, options exactes d'une Conditional Access policy). On ne réussit pas à 100 % en lecture pure.
2. **Apprenez à lire le portail Azure.** Les screenshots sont fréquents.
3. **KQL.** Pratiquez chaque jour 10 min sur le simulateur Sentinel ou Log Analytics demo.
4. **Pièges classiques** : RBAC vs Access Policies sur Key Vault, Service Endpoint vs Private Endpoint, NSG vs Azure Firewall, MDC vs Sentinel.
5. **Gérez le temps** : ~120 min pour ~40-60 questions + 1-2 case studies. ≈ 2 min/question.
6. **Marquez et passez** les questions floues au premier passage, revenez-y à la fin.
